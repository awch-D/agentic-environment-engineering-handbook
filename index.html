当模型能推理、调用工具并长时间执行时，工程瓶颈不再只是“该说什么”，而是：它在什么环境中观察、行动、验证、恢复与被治理。

01导读：工程对象正在外移

从“模型回答得好不好”，转向“模型在一套什么样的系统中工作”。

Working Definition

智能体环境工程（Agentic Environment Engineering）是对 Agent 工作环境的建模、合成、评估、应用与演化进行系统设计；在生产语境中，它进一步落到资源、接口、状态、隔离、反馈、预算和人机协同。

精确术语已出现在 2026 年中国科学院团队的综述预印本；相邻的 Agent Environment Engineering 也出现在清华大学计算机科学与技术系、智谱 AI 团队的 EurekAgent 论文与开源实现中。它们证明该概念已经进入研究讨论，但尚不是经过同行评审或由行业共同采用的标准。本文把它作为工作性总称，同时明确标注证据边界。S12–S14

一个实用判断是：若仅换模型或改提示词无法解决问题，而重新设计可见信息、工具契约、状态恢复、验证方式、预算或权限边界能够解决，那么瓶颈已经从模型调用转移到环境。S01–S03S08–S10

阅读边界：Prompt → Context → Harness → Loop → Environment 是教学性的“控制面外扩图”，不是业界共同确认、依次替代的五代历史。各层长期共存，Prompt 并未过时。
“环境”不是提示词的背景板；它是智能体可以感知、调用、改变并被约束的工作面。— 综合研究综述、官方工程文章与运行时实践

02概念地图：五个问题，五层工程

它们是控制面的逐层扩展，不是新词替换旧词。

PROMPT
提示工程 · Prompt Engineering：怎样编写、组织指令以获得目标行为？关注指令本身。S07
CONTEXT
上下文工程 · Context Engineering：推理时应策展并维护哪些 token，包括系统指令、工具、外部数据与历史？关注最小高信号信息集。S02
HARNESS
Harness Engineering · 中文译名未定：围绕模型建立 instructions、tools、routing、output requirements 与 validation checks 构成的完整契约。关注可执行、可验证的 Agent 外壳；已有官方实作，但术语边界仍宽。S09–S11
LOOP
Agent Loop / Loop Engineering：官方机制是“推理 → 工具调用 → 结果追加上下文 → 再推理”直到最终消息；Loop Engineering 则是更上层的早期个人框架，强调发现、分派、检查、记录和继续。S08S17–S18
ENVIRONMENT
智能体环境工程 · Agentic Environment Engineering：对资源、约束、接口及环境生命周期进行系统工程，让 Agent 能被训练、评测或在生产中长期工作。新兴研究术语 · S12–S14

“环境”至少有两种相邻但不可混同的含义：生产运行环境强调工具、文件系统、权限、Git、工件、预算、评估与人工控制；训练/评测环境还包括可合成、可交互的模拟世界。本文以生产运行环境为主，并在涉及 world model 时单独说明。S09S12–S16

03六层演进：从单次回答到可治理环境

关键节点不是线性替代，而是将新的控制面叠加在已有能力上。

04环境建模：定义 Agent 所处的世界

先说明 Agent 能看见什么、能做什么、世界如何变化和怎样反馈，再讨论工具与工作流。

Formal Model

综述将环境建模为 POMDP：状态空间 S、行动空间 A、状态转移 P、奖励 R、观察空间 Ω、观察函数 O 与折扣因子 γ。LLM Agent 根据多轮交互历史选择行动，环境返回新的观察和反馈。S12

八个属性轴

它们不是八种互斥环境，而是描述同一环境的八组坐标：

Symbolic ↔ Neural

状态转移来自显式规则或可执行代码，还是由神经网络世界模型近似。

Open-loop ↔ Closed-loop

Agent 按初始计划执行，还是根据每一步新观察持续调整行动。

Online ↔ Offline

与真实系统实时交互，还是只使用静态轨迹数据进行学习与比较。

MDP ↔ POMDP

Agent 能否访问完整真实状态，还是必须从不完整观察推断隐藏状态。

Deterministic ↔ Nondeterministic

相同状态与行动是否产生唯一结果，还是存在随机状态转移。

Discrete ↔ Continuous

行动来自有限可数集合，还是连续的实数向量空间。

Unimodal ↔ Multimodal

观察只包含单一模态，还是整合文字、图像、视频与其他传感信息。

Single-agent ↔ Multi-agent

环境转移由一个决策主体决定，还是取决于多个 Agent 的联合行动。

八个环境领域

领域典型交互面代表任务
GUI桌面、移动端、Web 界面导航、表单、跨应用操作
Deep Research搜索引擎、文档与多来源证据信息检索、多源推理、研究报告
Embodied机器人或 3D 虚拟世界空间导航、物理操作、长程规划
Game规则化开放世界与博弈系统解谜、社会推理、策略管理
Tool函数、API、数据库与 MCP工具选择、参数构造、结果反馈
Code代码、仓库、编译器和测试器生成、理解、验证、调试
Domain-specific医疗、科研、金融等专业系统领域决策与专门评测
Cross-domain跨异构任务和多种交互面通用性与迁移能力评估
建模边界：工具清单只是行动空间的一部分。完整环境模型还必须描述状态、转移、观察、反馈、可见性和多主体关系。

05环境合成：从手工场景到规模化生成

环境合成解决的不是“生成更多题目”，而是怎样构造可交互、可验证、可扩展的状态转移系统。

Symbolic Synthesis · 符号合成

用显式规则、可执行代码或形式化逻辑构建环境，优势是可解释、可复现和易验证。综述归纳为三条路径：S12

Task-driven

把数学、代码等高质量静态任务封装为可交互环境，复用既有数据和验证器。

Real-world-driven

将 Web、GUI、游戏等真实交互介质投影为可控虚拟环境，降低真实系统运行成本。

De novo

从少量种子甚至零样本直接生成全新环境，扩展自由度最高，但最依赖逻辑一致性和自动验证。

Neural Synthesis · 神经合成

用神经网络近似环境状态转移与观察生成，适合构造高维、开放和多模态世界：

Pixel-level

直接生成图像或视频观察，物理细节丰富，但长程一致性和计算成本是主要难点。

Word-level

以自然语言表达状态与变化，抽象程度高、成本较低，需控制幻觉和前后矛盾。

Latent-level

在学习到的表征空间中模拟语义与动力学，以效率换取可解释性和完整细节。

Design Trade-off

符号环境提供可靠性，神经环境提供生成规模。未来方向不是二选一，而是让显式约束与可学习世界模型形成 Neural–Symbolic Environment。S12S16

06环境评测：先证明世界可信，再测 Agent 能力

Agent 得分很高，不代表环境正确；弱验证器、任务泄漏或失真动力学都可能制造虚假能力。

Correctness · 正确性

状态转移是否合法、任务是否可解、验证器是否可靠。常用程序执行、单元测试、可解性检查和轨迹验证。

Diversity · 多样性

任务、状态、工具和语言空间是否覆盖广泛且不重复，避免 Agent 只记住少量模式。

Complexity · 复杂度

难度是否既非过易也非不可完成，可用步骤数、工具数、代码量或人类/强模型表现校准。

Fidelity · 保真度

合成环境是否忠实反映真实系统、物理动力学或用户行为；神经环境还需评估感知与交互一致性。

对象重点证据典型风险
符号环境可执行性、规则一致、可解性、验证器正确规则漏洞、测试泄漏、奖励投机
神经环境语义一致、交互一致、动态演化和物理真实性长程漂移、不可控生成、观察失真
不要混淆两类评估：这里评的是“环境是否可信”；后文运行评估关注“Agent 在既定环境中是否完成任务、轨迹是否合规”。两者必须同时成立。S12

07协同演化：Agent 与环境相互抬高上限

环境不只是固定考场；它可以积累经验、调整工作流、生成训练轨迹并持续改变难度。

Agent Evolution · 四条能力演化路径

Memory-centric

从实例轨迹、抽象脚本到结构化技能，将交互经验沉淀为可检索、可调用的能力资产。

Orchestration-centric

从固定工作流到自动编排，再到可持续演化的工作流结构。

Trajectory-centric Offline

通过任务合成、轨迹合成与轨迹精炼，生成可用于模仿学习和离线优化的数据。

Exploration-centric Online

利用在线探索、奖励塑形和算法优化，让 Agent 根据实时环境反馈更新策略。

Environment Evolution · 三条环境演化路径

路径演化机制主要目标
Neural-drivenSelf-play 或 world model 学习环境参数与未来状态生成更丰富、更适应 Agent 的交互场景
Difficulty-driven显式课程或根据实时表现隐式调节难度让任务保持在可学习但具有挑战的区间
Scaling-driven扩大既有场景多样性，或引入全新结构与模态推动跨场景泛化和更通用的能力
Environment Eₜ ──任务 / 观察 / 奖励──► Agent πₜ ▲ │ │ ▼ 环境生成 / 难度 / 扩展 ◄──轨迹 / 技能 / 策略── Agent πₜ₊₁

真正的共演化必须保留外部验证：环境不能只迎合当前 Agent，Agent 也不能通过修改奖励或验证器伪造进步。S12

08环境原语：Agent 依赖的七类条件

不是“给模型更多工具”，而是设计每一次行动都能被理解、限制和复现的工作面。

任务 / 事件 │ ▼ ┌─────────── Environment Contract ────────────┐ │ Context · State · Tools · Sandbox · Policy │ │ Evidence · Evaluation · Human Escalation │ └────────────────────┬────────────────────────┘ ▼ Agent Runtime │ ▼ Observable World Change

01 · 任务事实

当前目标、约束、输入来源和成功标准。没有可判定目标的自主循环只会把不确定性放大。

02 · 上下文装配

按任务阶段动态挑选文件、记忆、检索证据与最近观察；不是把所有历史消息机械塞进窗口。S02

03 · 状态与恢复

把可继续执行的信息放到模型窗口之外：检查点、执行游标、工件引用、失败原因和待人工决定项。

04 · 工具契约

每个工具都有明确的输入、输出、权限、幂等性、超时、成本与可观察结果；工具是环境 API，不是提示词附注。S04

05 · 执行隔离

将文件系统、网络、密钥、生产操作和并发任务限定在合适的沙箱、工作区或审批边界内。

06 · 证据与评估

用测试、结构化校验、外部观测或独立评审判断行动结果；不要把“模型说完成了”视为证据。S03

07 · 治理与接管

定义预算、停止条件、风险动作的批准人、撤销路径和升级机制，让人类进入必要的决策点而不是每一步。

贯穿原则 · 可追溯

任何重要输出都应回溯到输入、工具调用、环境快照、验证结果与授权记录，才能调试、审计和改进。

工具契约比工具数量更重要

在 demo 中，工具只是函数列表；在生产环境中，工具是具备副作用的能力接口。优先补齐模式校验、最小权限、超时、重试语义、关联 ID、结果出处和审计事件，再讨论让 Agent 调更多工具。MCP 等开放协议能降低接入摩擦，但协议本身并不替代业务权限设计。S04

09运行时设计：把“对话”变成可恢复的任务

长程 Agent 的最小单位不是聊天会话，而是带状态、预算和证据的任务执行。

一次真实任务至少经历：接收事件、确认目标、装配上下文、选择或调用能力、写入状态、验证结果、决定继续或退出。运行时的职责是让这条链路在超时、模型切换、工具失败和人工接管后仍能保持一致。

状态层应该保存什么不应该保存什么典型载体
Working当前子目标、步骤游标、临时工具结果所有历史原文短期任务对象 / checkpoint
Durable已确认事实、工件版本、待办和恢复指针未核验的模型猜测数据库 / 状态文件 / issue
Evidence测试输出、diff、外部观测、审批记录只有“成功”布尔值的摘要trace、日志、制品库
Design Rule

状态不是聊天记录。聊天记录服务于对话连续性；任务状态服务于可恢复性、并发控制和可信交接。两者可以相互引用，但不应互相替代。

这也是 Context Engineering 与 Environment Engineering 的分界之一：前者解决“下一次调用给模型什么信息”，后者还必须解决“该信息从何而来、是否可信、谁能改、失败后如何恢复”。

10环境闭环：让行动受证据约束

Loop 不是 while true；它是一组带预算、观测、验证与退出条件的控制结构。

[Intent] → [Plan] → [Act] → [Observe] → [Assess] ▲ │ │ │ ▼ ├── pass → commit / report └──── feedback + state ← [Evidence] ├── fail → revise / retry └── uncertain → human review

最小循环的价值在于让模型的行为与环境反馈耦合：工具返回的客观数据、测试输出、外部系统状态或用户批准，比模型对自己的叙述更可信。复杂流程可以嵌套多个循环，但每一层都应有独立的资源上限和退出理由。S01

四种常用控制模式

模式适合什么关键设计点
Workflow步骤与规则稳定的流程把路径写死；模型只处理不确定的小块。
Agent Loop需要基于观察动态选工具的任务限制步数、工具范围、预算和可行动作。
Evaluator–Optimizer可生成候选、可定义质量标准的工作让评估基于 rubric、测试或外部事实，而不只靠同一模型的自评。
Event-driven由工单、代码变更、消息或数据到达触发事件应幂等、可重放；状态和去重键必须在模型外保存。

“Maker / Checker”应被理解为证据分离

常见的说法是“生成者不能给自己验收”。更精确的工程表述是：产生行动的通道不应独占成功判定。Checker 可以是单元测试、schema 校验、独立模型、人工审核或外部系统回执；是否需要不同模型，取决于风险和可用的客观证据。S01S03

  • 成功：满足可观察的完成标准，写入工件与证据。
  • 可重试：收到具体失败原因，且下一次尝试有新的信息或策略。
  • 需升级：连续无进展、权限不足、目标歧义或高影响操作。
  • 强制停止：预算耗尽、超过时间窗口、检测到风险或用户撤销。

11控制与安全：把权限做成环境属性

模型能力越强，越不能把“请小心”当成安全机制。

环境工程的安全目标不是保证模型永不犯错，而是让错误难以越权、容易发现、能够撤销。因此，权限应绑定到任务、工作区、身份、时效和工具，而不是只埋在系统提示词中。

只读与低影响

允许自动检索、分析、草拟与本地模拟；记录数据来源和访问范围。

可逆写入

允许在隔离分支、草稿箱或临时环境修改；要求 diff、检查点和明确回滚路径。

高影响动作

如提交生产变更、对外发送、删除或资金相关操作，应要求明确的人类审批与最小范围执行。

禁止自动化

高风险且不可逆、缺乏充分证据或授权不清的操作，不应交给无人值守 Agent。

把外部内容视作不可信数据

网页、邮件、文档和工具返回值可能携带与用户目标无关的操控性文本。环境应在数据层区分“待处理内容”和“可执行指令”，限制工具调用的能力范围、要求敏感动作出示证据，并记录 Agent 为什么采取某一步。这种分层比单纯要求模型“忽略恶意提示”更可靠。防御原则

安全边界：沙箱、最小权限和审批并不是可选的“企业功能”。只要 Agent 拥有写入、联网、访问机密或代表用户行动的能力，它们就是环境设计的核心组成。

12运行评估与观测：没有测量，就没有可靠改进

把一次 Agent 任务当作可检查的“episode”，而不是不可解释的一段聊天。

评估不是上线前的一次考试。长程 Agent 需要三层反馈：离线任务集验证版本是否退化，在线 trace发现真实失败模式，结果回流将错误归因到模型、上下文、工具、策略、权限还是环境状态。S03

层级要回答的问题可用证据
Outcome任务结果真的满足用户目标吗？测试、业务指标、人工验收、外部系统回执
Trajectory过程是否高效、合规且没有走偏？trace、工具调用序列、预算、策略事件
Environment失败是否由工具契约、上下文供给或运行时造成？环境快照、版本、权限决策、状态转移

LLM-as-a-judge 可以是有效工具，但不是免检的真相机。为它提供明确 rubric、用人工标注样本做校准、避免让它审自己参与生成的内容，并保留可解释的硬指标。可测任务优先使用确定性检查；主观任务再用组合式评审。S03

13落地路线：先造可信闭环，再延长自主性

不要从“全自动 Agent”开始；从一个有真实结果、低风险且可自动验证的任务开始。

  1. 选定一个窄任务目标明确、输入边界清楚、结果可以被测试或人工快速判断。例如：整理结构化数据、生成草稿、修复低风险缺陷、检查配置偏差。
  2. 先写环境契约明确哪些数据可信、可调的工具有哪些、每个动作的权限、成功证据、失败后的状态与升级人。
  3. 把状态和证据移出聊天建立任务 ID、工件位置、检查点和 trace；确保中断后可以解释和恢复，而不是重新问模型“刚才做到哪了”。
  4. 设置硬护栏为调用次数、token、时长、网络范围、写入范围和并发数设预算；高影响动作默认要求审批。
  5. 建立最小评估集收集真实任务、失败案例和边界输入；每次变更模型、上下文、工具或策略后都回归测试。
  6. 再扩展自主性只有在错误可定位、过程可观察、失败可恢复时，才逐步加入重试、多 Agent、事件触发和更长的运行周期。
Adoption Heuristic

自治等级应由可验证性决定，而不是由模型演示效果决定。越难独立判定“做对了没有”，越应缩小 Agent 的权限和行动跨度。

14边界与误解：别让新名词掩盖旧问题

以下说法看似合理，实际会让团队跳过最重要的工程工作。

  • “环境工程就是接一个 RAG。”检索只是上下文供给的一种方式；状态、工具、隔离、评估和治理同样属于环境。
  • “上下文越多，Agent 越聪明。”上下文有成本、时效和噪声。正确目标是每个决策时刻拥有足够且可信的信息，而不是最大化 token 占用。
  • “多 Agent 天然更可靠。”更多 Agent 也意味着更多状态同步、成本、权限面和失败模式。只有当分工和独立证据确实带来价值时才并行化。
  • “沙箱解决了所有安全问题。”沙箱能限制执行面，但不能替代数据最小化、身份授权、审批、审计、工具输入校验和业务规则。
  • “让模型自评即可完成闭环。”自评可提供信号，却不等同于外部证据。优先把成功判定落在测试、约束、系统回执或独立审核上。
  • “智能体环境工程已经是统一标准。”不是。本文用它组织跨领域实践；发布和决策时应回到具体的协议、产品文档、论文和风险模型。
